PDFRecon Forensic Analysis Manual

Purpose: This manual explains how to manually verify each forensic indicator that PDFRecon detects. Use these techniques to confirm automated findings or conduct independent analysis.

Classification System

Classification Meaning Action Required
YES Strong evidence of alteration Immediate investigation
FOUND Technical indicators detected Careful review
NOT DETECTED No known indicators Lower priority
⚠️ Important: Automated detection is a starting point. Always verify findings manually before drawing conclusions. Some indicators can result from legitimate operations.

About File System Timestamps

The 'File Created' and 'File Modified' columns show timestamps from your operating system. These are unreliable for forensic purposes:

  • Copying a file updates the creation date
  • Moving between drives can reset timestamps
  • Cloud sync services often replace original dates
  • Timestamps can be easily manipulated

Always use Show Timeline to examine metadata stored inside the PDF file.

Om filsystemets tidsstempler

Kolonnerne 'File Created' og 'File Modified' viser tidsstempler fra dit operativsystem. Disse er upålidelige til retsmedicinske formål:

  • Flytning mellem drev kan nulstille tidsstempler
  • Cloud-synkroniseringstjenester erstatter ofte originale datoer
  • Tidsstempler kan nemt manipuleres

Timestamp Spoofing HIGH RISK

What it means: The file system creation date is significantly older than the internal PDF creation date. This is physically impossible under normal circumstances and strongly indicates the system clock was rolled back or the file timestamp was artificially manipulated (timestomped).

Step-by-Step Verification

1 Right-click the file in Windows -> Properties -> Details. Note the "Date created".
2 Use ExifTool or open the PDF in a hex editor and search for /CreationDate.
3 Compare the timestamps. If the external Date Created is older than the internal /CreationDate, the file has been tampered with.

Spoofing med tidsstempel HØJ RISIKO

Hvad betyder det: Filsystemets oprettelsesdato er væsentligt ældre end den interne PDF-oprettelsesdato. Dette er fysisk umuligt under normale omstændigheder og indikerer kraftigt, at systemuret blev rullet tilbage, eller at filens tidsstempling blev kunstigt manipuleret (tidsstemplet).

Trin-for-trin-bekræftelse

1 Højreklik på filen i Windows -> Egenskaber -> Detaljer. Bemærk "Oprettelsesdatoen".
2 Brug ExifTool eller åbn PDF'en i en hex-editor og søg efter /CreationDate.
3 Sammenlign tidsstemplerne. Hvis den eksterne Oprettelsesdato er ældre end den interne /CreationDate, er filen blevet manipuleret.

Phishing Directives HIGH RISK

What it means: The PDF contains actions that can submit form data to an external URL or launch external files/applications.

Manual Verification

1 Search for /Action dictionaries in the PDF content.
2 Look for /S /SubmitForm and check the /F key for target URLs.
3 Look for /S /Launch and check for automated execution of cmd.exe or similar commands.

Phishing-direktiver HØJ RISIKO

Hvad betyder det: PDF'en indeholder handlinger, der kan sende formulardata til en ekstern URL eller starte eksterne filer/applikationer.

Manuel verifikation

1 Søg efter /Handling ordbøger i PDF-indholdet.
2 Se efter /S /SendForm og tjek /F nøgle til mål-URL'er.
3 Se efter /S /Lancering og kontroller for automatiseret udførelse af cmd.exe eller lignende kommandoer.

TouchUp_TextEdit HIGH RISK

What it means: Adobe Acrobat's TouchUp Text tool was used to manually edit text in the PDF. This leaves specific metadata that proves direct content modification.

Step 1: Find the TouchUp Marker

1 Open the PDF in a hex editor (HxD, 010 Editor)
2 Press Ctrl+F and search for text: TouchUp_TextEdit
3 Alternative: Search for hex bytes:
2F 54 6F 75 63 68 55 70 5F 54 65 78 74 45 64 69 74 = /TouchUp_TextEdit

Step 2: Examine the PieceInfo Structure

When found, the surrounding structure looks like:

/PieceInfo <<
  /AdobePhotoshop <<
    /LastModified (D:20240115143000+01'00')
    /Private <<
      /TouchUp_TextEdit true
    >>
  >>
>>
Document these values:
  • Byte offset where /TouchUp_TextEdit found
  • The /LastModified timestamp
  • Screenshot of hex context

Step 3: Extract the Edited Text (Advanced)

Understanding the Challenge: TouchUp edits are stored in content streams using CID (Character ID) encoding. The text you see in PDFRecon's output (e.g., "ZK,KZK,ZP=GZ") appears garbled because it needs to be decoded through the font's character mapping dictionary.

How CID Encoding Works

When Adobe TouchUp edits text, it:

  1. Creates new font subsets (e.g., /C2_0, /C2_1)
  2. Stores text as CID numbers, not actual letters
  3. Maps CIDs to Unicode via a /ToUnicode CMap

Step 3a: Find the Font Reference

1 In the content stream, look for font selections like: 
/C2_0 12 Tf    ← Using font "C2_0" at 12pt
<0047004800490048> Tj    ← CID-encoded text
2 Note the font name (e.g., C2_0) - this is the key to decoding

Step 3b: Find the Font Dictionary

3 Search for /C2_0 in the PDF to find its font dictionary: 
/C2_0 <<
  /Type /Font
  /Subtype /Type0
  /BaseFont /BCDFGH+Arial-BoldMT
  /Encoding /Identity-H
  /DescendantFonts [15 0 R]
  /ToUnicode 16 0 R     ← This object contains the mapping!
>>

Step 3c: Find the ToUnicode CMap

4 Go to the referenced object (e.g., object 16) and find the CMap: 
16 0 obj
<</Length 500>>
stream
/CIDInit /ProcSet findresource begin
12 dict begin
begincmap
/CMapName /Adobe-Identity-UCS def
/CIDSystemInfo <<
  /Registry (Adobe)
  /Ordering (UCS)
  /Supplement 0
>> def
1 begincodespacerange
<0000> <FFFF>
endcodespacerange
15 beginbfchar
<0003> <0020>    ← CID 0003 = Unicode 0020 (space)
<0047> <0041>    ← CID 0047 = Unicode 0041 (letter A)
<0048> <0042>    ← CID 0048 = Unicode 0042 (letter B)
<0049> <0043>    ← CID 0049 = Unicode 0043 (letter C)
...
endbfchar
endcmap
endstream
endobj

Step 3d: Decode the Text Manually

5 Now decode the hex string <0047004800490048>:
Hex CID From CMap Unicode Character
0047 <0047> <0041> U+0041 A
0048 <0048> <0042> U+0042 B
0049 <0049> <0043> U+0043 C
0048 <0048> <0042> U+0042 B

Result: The text is "ABCB"

Why PDFRecon Shows Garbled Text

When PDFRecon extracts TouchUp text, it attempts to decode using available font mappings. If the ToUnicode CMap is missing, incomplete, or uses non-standard encoding, the result appears as random characters.

Common patterns:

  • ZK,KZK - CID values interpreted as ASCII
  • /IZ/IZK?ZGK - Mix of operators and CID data
  • Numbers like 2710832 - Successfully decoded numeric text

When you see partial readable text mixed with gibberish, the readable parts are text that could be decoded, while gibberish indicates missing mappings.

Quick Verification Commands

# Search for TouchUp marker
grep -l "TouchUp_TextEdit" *.pdf

# Find all font references in PDF
grep -oP '/C\d+_\d+' file.pdf | sort | uniq

# Decompress for analysis
qpdf --qdf --object-streams=disable input.pdf readable.pdf

# Then search in readable version
grep -A 20 "ToUnicode" readable.pdf

TouchUp_TextEdit HØJ RISIKO

Hvad betyder det: Adobe Acrobats TouchUp Text-værktøj blev brugt til manuelt at redigere tekst i PDF. Dette efterlader specifikke metadata, der beviser direkte indholdsændring.

Trin 1: Find TouchUp-markøren

1 Åbn PDF'en i en hex-editor (HxD, 010 Editor)
2 Trykke Ctrl+F og søg efter tekst: TouchUp_TextEdit
3 Alternativ: Søg efter hex bytes:
2F 54 6F 75 63 68 55 70 5F 54 65 78 74 45 64 69 74 = /TouchUp_TextEdit

Trin 2: Undersøg PieceInfo-strukturen

Når den er fundet, ser den omgivende struktur ud som:

/PieceInfo <<
  /AdobePhotoshop <<
    /LastModified (D:20240115143000+01'00')
    /Privat <<
      /TouchUp_TextEdit true
    >>
  >>
>>
Dokumenter disse værdier:
  • Byte offset hvor /TouchUp_TextEdit fundet
  • De /LastModified tidsstempel
  • Skærmbillede af hex-kontekst

Trin 3: Udpak den redigerede tekst (avanceret)

Forstå udfordringen: TouchUp-redigeringer gemmes i indholdsstrømme ved hjælp af CID (Tegn ID) kodning. Den tekst, du ser i PDFRecons output (f.eks. "ZK,KZK,ZP=GZ") ser forvansket ud fordi den skal afkodes gennem skrifttypens tegnkortordbog.

Sådan fungerer CID-kodning

Når Adobe TouchUp redigerer tekst, gør det:

  1. Opretter nye skrifttypeundersæt (f.eks. /C2_0, /C2_1)
  2. Gemmer tekst som CID-numre, ikke egentlige bogstaver
  3. Maps CID'er til Unicode via en /ToUnicode CMap

Trin 3a: Find skrifttypereferencen

1 I indholdsstrømmen skal du se efter skrifttypevalg som: 
/C2_0 12 Tf ← Brug af font "C2_0" ved 12pt
<0047004800490048> Tj ← CID-kodet tekst
2 Bemærk skrifttypenavnet (f.eks. C2_0) - dette er nøglen til afkodning

Trin 3b: Find skrifttypeordbogen

3 Søg efter /C2_0 i PDF'en for at finde dens skrifttypeordbog: 
/C2_0 <<
  /Skriv /Skrifttype
  /Undertype /Type0
  /BaseFont /BCDFGH+Arial-BoldMT
  /Kodning /Identitet-H
  /DescendantFonts [15 0 R]
  /ToUnicode 16 0 R ← Dette objekt indeholder kortlægningen!
>>

Trin 3c: Find ToUnicode CMap

4 Gå til det refererede objekt (f.eks. objekt 16) og find CMap: 
16 0 obj
</Længde 500>>
stream
/CIDInit /ProcSet findresource begynder
12 dict begynder
startcmap
/CMapName /Adobe-Identity-UCS def
/CIDSystemInfo <<
  /Registry (Adobe)
  /Bestilling (UCS)
  /Tillæg 0
>> def
1 startkodemellemrum
<0000> <FFFF>
endcodespacerange
15 begynderbfchar
<0003> <0020> ← CID 0003 = Unicode 0020 (mellemrum)
<0047> <0041> ← CID 0047 = Unicode 0041 (bogstav A)
<0048> <0042> ← CID 0048 = Unicode 0042 (bogstav B)
<0049> <0043> ← CID 0049 = Unicode 0043 (bogstav C)
...
endbfchar
endcmap
endstream
endobj

Trin 3d: Afkode teksten manuelt

5 Afkod nu hex-strengen <0047004800490048>:
Hex CID Fra CMap Unicode Karakter
0047 <0047> <0041> U+0041 A
0048 <0048> <0042> U+0042 B
0049 <0049> <0043> U+0043 C
0048 <0048> <0042> U+0042 B

Resultat: Teksten er "ABCB"

Hvorfor PDFRecon viser forvansket tekst

Når PDFRecon udtrækker TouchUp-tekst, forsøger den at afkode ved hjælp af tilgængelige skrifttypetilknytninger. Hvis ToUnicode CMap mangler, er ufuldstændig eller bruger ikke-standard kodning, resultatet vises som tilfældigt tegn.

Fælles mønstre:

  • ZK,KZK - CID-værdier fortolket som ASCII
  • /IZ/IZK?ZGK - Blanding af operatører og CID-data
  • Tal som 2710832 - Succesfuld afkodet numerisk tekst

Når du ser delvis læsbar tekst blandet med volapyk, er de læsbare dele tekst, der kunne være afkodet, mens volapyk indikerer manglende kortlægninger.

Hurtige verifikationskommandoer

# Søg efter TouchUp-markør
grep -l "TouchUp_TextEdit" *.pdf

# Find alle skrifttypereferencer i PDF
grep -oP '/C\d+_\d+' file.pdf | sortere | enestående

# Dekomprimer til analyse
qpdf --qdf --object-streams=deaktiver input.pdf readable.pdf

# Søg derefter i læsbar version
grep -A 20 "ToUnicode" readable.pdf

Has Revisions HIGH RISK

What it means: The PDF contains previous versions saved incrementally. PDFRecon extracted hidden revisions proving the document was modified after initial creation.

Step 1: Count %%EOF Markers

1 In hex editor, search for %%EOF
25 25 45 4F 46 = %%EOF
2 Count occurrences and note byte offsets:
  • 1 occurrence = Original document (no revisions)
  • 2+ occurrences = Document has revisions

Step 2: Extract Revisions Manually

Each %%EOF marks the end of a complete PDF version:

Bytes 0 → first %%EOF  = Revision 1 (oldest)
Bytes 0 → second %%EOF = Revision 2
Bytes 0 → third %%EOF  = Revision 3
...
Bytes 0 → last %%EOF   = Current version
3 In HxD:
  • Note the byte offset of first %%EOF (e.g., 45000)
  • Edit → Select Block → Start: 0, End: 45004 (offset + 5)
  • Copy and paste into new file
  • Save as filename_rev1.pdf

Step 3: Follow the /Prev Chain

Each incremental save has a trailer pointing to the previous version:

trailer
<<
  /Size 150
  /Root 1 0 R
  /Prev 12345      ← Byte offset of previous xref
>>
startxref
45000
%%EOF
4 Search for /Prev followed by a number. Each /Prev value points to an earlier cross-reference table.

Command Line Extraction

# Find all %%EOF positions
grep -boa "%%EOF" file.pdf

# Output example:
# 45000:%%EOF
# 98500:%%EOF
# 125000:%%EOF

# Extract first revision (oldest)
head -c 45005 file.pdf > revision1.pdf

# Extract second revision
head -c 98505 file.pdf > revision2.pdf

# Count revisions
grep -c "%%EOF" file.pdf
Important Note: Incremental updates are a NORMAL PDF feature. They occur during:
  • Digital signing (required to preserve signature)
  • Form filling
  • Adding comments/annotations
  • Normal "Save" in Adobe Acrobat
Compare the CONTENT between revisions to assess if changes are suspicious.

Har revisioner HØJ RISIKO

Hvad betyder det: PDF'en indeholder tidligere versioner, der er gemt trinvist. PDFRecon udtrak skjulte revisioner, der beviser, at dokumentet blev ændret efter den første oprettelse.

Trin 1: Tæl %%EOF-markører

1 I hex-editor, søg efter %%EOF
25 25 45 4F 46 = %%EOF
2 Tæl forekomster og notebyteforskydninger:
  • 1 forekomst = Originaldokument (ingen revisioner)
  • 2+ forekomster = Dokumentet har revisioner

Trin 2: Udpak revisioner manuelt

hver %%EOF markerer slutningen på en komplet PDF-version:

Bytes 0 → første %%EOF = Revision 1 (ældste)
Bytes 0 → sekund %%EOF = Revision 2
Bytes 0 → tredje %%EOF = Revision 3
...
Bytes 0 → sidste %%EOF = Aktuel version
3 I HxD:
  • Bemærk byte offset af først %%EOF (f.eks. 45.000)
  • Rediger → Vælg blok → Start: 0, Slut: 45004 (offset + 5)
  • Kopiér og indsæt i ny fil
  • Gem som filnavn_rev1.pdf

Trin 3: Følg /Prev Chain

Hver trinvis lagring har en trailer, der peger på den tidligere version:

trailer
<<
  /Størrelse 150
  /Root 1 0 R
  /Forrige 12345 ← Byte offset af forrige xref
>>
startxref
45.000
%%EOF
4 Søg efter /Forrige efterfulgt af et tal. hver /Forrige værdi peger på en tidligere krydsreferencetabel.

Kommandolinjeudtrækning

# Find alle %%EOF-stillinger
grep -boa "%%EOF" fil.pdf

# Output eksempel:
# 45000:%%EOF
# 98500:%%EOF
# 125000:%%EOF

# Uddrag første revision (ældste)
head -c 45005 file.pdf > revision1.pdf

# Uddrag anden revision
head -c 98505 file.pdf > revision2.pdf

# Tæl revisioner
grep -c "%%EOF" fil.pdf
Vigtig bemærkning: Inkrementelle opdateringer er en NORMAL PDF-funktion. De opstår under:
  • Digital signering (påkrævet for at bevare signaturen)
  • Formular udfyldning
  • Tilføjelse af kommentarer/annoteringer
  • Normal "Gem" i Adobe Acrobat
Sammenlign INDHOLDET mellem revisioner for at vurdere, om ændringer er mistænkelige.

JavaScript Auto-Execute HIGH RISK

What it means: The PDF executes JavaScript automatically when opened, potentially modifying displayed content dynamically.

Step 1: Search for OpenAction

1 Search for /OpenAction
2F 4F 70 65 6E 41 63 74 69 6F 6E = /OpenAction

Step 2: Examine What It Triggers

Direct JavaScript:

/OpenAction <<
  /S /JavaScript
  /JS (app.alert\("Warning"\);)
>>

Indirect reference:

/OpenAction 15 0 R

Then find object 15 to see the JavaScript code.

Step 3: Find and Decode JavaScript

JavaScript can be stored as:

  • Direct string: /JS (javascript code here)
  • Hex string: /JS <6170702E616C657274>
  • Stream reference: /JS 20 0 R

Decoding Hex JavaScript

Hex:   61 70 70 2E 61 6C 65 72 74
ASCII: a  p  p  .  a  l  e  r  t

Step 4: Check Additional Actions (AA)

Search for /AA which defines actions on various triggers:

/AA <<
  /O <</S /JavaScript /JS (...)>>    % On page open
  /C <</S /JavaScript /JS (...)>>    % On page close
  /E <</S /JavaScript /JS (...)>>    % On mouse enter
  /X <</S /JavaScript /JS (...)>>    % On mouse exit
>>

JavaScript automatisk udførelse HØJ RISIKO

Hvad betyder det: PDF'en udfører muligvis JavaScript automatisk, når den åbnes ændring af vist indhold dynamisk.

Trin 1: Søg efter OpenAction

1 Søg efter /OpenAction
2F 4F 70 65 6E 41 63 74 69 6F 6E = /OpenAction

Trin 2: Undersøg, hvad det udløser

Direkte JavaScript:

/OpenAction <<
  /S /JavaScript
  /JS (app.alert\("Advarsel"\);)
>>

Indirekte reference:

/OpenAction 15 0 R

Find derefter objekt 15 for at se JavaScript-koden.

Trin 3: Find og afkod JavaScript

JavaScript kan gemmes som:

  • Direkte streng: /JS (javascript-kode her)
  • Hex streng: /JS <6170702E616C657274>
  • Stream reference: /JS 20 0 R

Afkodning af Hex JavaScript

Hex: 61 70 70 2E 61 6C 65 72 74
ASCII: a p p.  a l e r t

Trin 4: Tjek yderligere handlinger (AA)

Søg efter /AA som definerer handlinger på forskellige udløsere:

/AA <<
  /O <</S /JavaScript /JS (...)>> % På side åben
  /C <</S /JavaScript /JS (...)>> % På sidelukning
  /E <</S /JavaScript /JS (...)>> % Ved indtastning af mus

>>

Hidden Annotations MEDIUM RISK

What it means: The PDF contains annotations with the 'Hidden' or 'Invisible' flags set, possibly to obscure content.

Manual Verification

1 Search for /Annots arrays in the PDF document.
2 Inspect individual annotation objects (/Type /Annot) for the /F (Flags) key.
3 Check if Bit 2 (Hidden, value 2) or Bit 1 (Invisible, value 1) is set in the /F integer.

Skjulte anmærkninger MIDDEL RISIKO

Hvad betyder det: PDF'en indeholder annoteringer med "Skjult" eller "Usynlig" flag sat, muligvis for at skjule indhold.

Manuel verifikation

1 Søg efter /Annoter arrays i PDF-dokumentet.
2 Undersøg individuelle anmærkningsobjekter (/Skriv /Annot) for /F (Flag) tast.
3 Kontroller, om bit 2 (skjult, værdi 2) eller bit 1 (usynlig, værdi 1) er indstillet i /F heltal.

Missing Objects HIGH RISK

What it means: The PDF references objects that don't exist in the file, indicating corruption or improper editing.

Step 1: List All Object Definitions

Objects are defined as X Y obj:

5 0 obj        ← Object 5, generation 0
<</Type /Page>>
endobj

Step 2: List All Object References

References use format X Y R:

/Kids [3 0 R 5 0 R 7 0 R]    ← References objects 3, 5, and 7

Step 3: Compare Lists

# List all definitions
grep -oP '\d+(?= 0 obj)' file.pdf | sort -n | uniq > defined.txt

# List all references
grep -oP '\d+(?= 0 R)' file.pdf | sort -n | uniq > referenced.txt

# Find missing (referenced but not defined)
comm -23 referenced.txt defined.txt

Any object in referenced.txt not in defined.txt is MISSING.

Manglende objekter HØJ RISIKO

Hvad betyder det: PDF'en refererer til objekter, der ikke findes i filen, hvilket angiver korruption eller ukorrekt redigering.

Trin 1: Vis alle objektdefinitioner

Objekter er defineret som X Y obj:

5 0 obj ← Objekt 5, generation 0
</Type /Page>>
endobj

Trin 2: Vis alle objektreferencer

Referencer bruger format X Y R:

/Børn [3 0 R 5 0 R 7 0 R] ← Referencer objekter 3, 5 og 7

Trin 3: Sammenlign lister

# Liste over alle definitioner
grep -oP '\d+(?= 0 obj)' file.pdf | sortere -n | uniq > defineret.txt

# Liste over alle referencer
grep -oP '\d+(?= 0 R)' file.pdf | sortere -n | uniq > referenced.txt

# Find mangler (refereret, men ikke defineret)
comm -23 referenced.txt defined.txt

Enhver genstand i refereret.txt ikke i defineret.txt mangler.

Multiple Font Subsets MEDIUM

What it means: The same font style is embedded multiple times with different subsets, suggesting text was added at different times or with different tools.

How to Identify

Font subsets have the format: XXXXXX+FontName

/BaseFont /AAAAAA+Arial-Regular     ← First subset
/BaseFont /BBBBBB+Arial-Regular     ← SAME font, different prefix = SUSPICIOUS
/BaseFont /CCCCCC+Arial-Bold        ← Different style = NORMAL

Manual Search

# Find all font subsets
grep -oP '/BaseFont /[A-Z]{6}\+\S+' file.pdf | sort | uniq

# Count unique subsets per font family
grep -oP '/BaseFont /[A-Z]{6}\+(\S+)' file.pdf | \

Multiple startxref MEDIUM

What it means: Multiple cross-reference tables exist, indicating incremental saves.

How to Find

# Or show values

Understanding the Structure

startxref
12345       ← Byte offset of this version's xref table
%%EOF

... (new content) ...

startxref
45678       ← Byte offset of newer xref table
%%EOF

Following the Chain

  1. Go to the byte offset indicated by startxref
  2. Find /Prev XXXXX in the trailer
  3. Go to that offset for the previous xref
  4. Repeat until no more /Prev entries

Digital Signature MEDIUM

What it means: Document was digitally signed. A broken/invalid signature indicates modification after signing.

Step 1: Find Signature Object

Search for /Type /Sig:

/Type /Sig
/Filter /Adobe.PPKLite
/SubFilter /adbe.pkcs7.detached
/ByteRange [0 1000 5000 10000]
/Contents <308204...>

Step 2: Understand ByteRange

/ByteRange [start1 len1 start2 len2]

  • Bytes 0-999 (first 1000 bytes) are signed
  • Bytes 1000-4999 contain the signature (excluded)
  • Bytes 5000-14999 are signed

If there are %%EOF markers AFTER the signature's ByteRange ends, the document was modified after signing!

Format: D:YYYYMMDDHHmmSS±HH'mm'

Format: YYYY-MM-DDTHH:mm:SS±HH:MM

2024-01-15 = Date

Cross-Reference Table Reference

Standard XRef Format

xref
0 6                          ← Start at object 0, 6 entries
0000000000 65535 f           ← Object 0: free, gen 65535
0000000017 00000 n           ← Object 1: at byte 17, gen 0
0000000081 00000 n           ← Object 2: at byte 81
0000000000 00001 f           ← Object 3: deleted (free), gen 1
0000000331 00000 n           ← Object 4: at byte 331
0000000409 00000 n           ← Object 5: at byte 409

trailer
<<
  /Size 6                    ← Total object count
  /Root 1 0 R                ← Document catalog
  /Info 5 0 R                ← Info dictionary
  /ID [<abc> <def>]         ← Document IDs
  /Prev 12345                ← Previous xref (incremental)
>>
startxref
500                          ← Byte offset of this xref
%%EOF

Entry Format

OOOOOOOOOO GGGGG n/f

  • OOOOOOOOOO = 10-digit byte offset
  • GGGGG = 5-digit generation number
  • n = in use, f = free (deleted)

10 0 obj
<<
  /Type /XRef
  /Size 100
  /W [1 3 1]          ← Column widths in bytes
  /Root 1 0 R
  /Info 5 0 R
>>
stream
[binary data]
endstream
endobj
